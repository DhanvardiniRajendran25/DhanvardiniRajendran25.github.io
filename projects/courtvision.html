Overview

One assistant for the whole coaching prep loop.

CourtVision AI is a natural-language basketball scouting and live game-simulation assistant I built and shipped as the product manager. A coach asks a plain question about an opponent, gets a grounded, sourced read, hands that read straight into an interactive game simulation, and then makes live tactical calls that actually change the outcome. It runs as three Gemini agents that pass intelligence to each other, deployed live on Cloud Run inside a Google Frontiers Hackathon window.

The product bet is simple: the value is not three separate AI toys, it is the hand-off between them. A weakness the Scout finds flows into the game the Simulator runs, so the plan is tested, not guessed. For the full requirement set, read the complete PRD.

The problem

Coaching intelligence lives in three places at once.

Coaches spend hundreds of hours preparing: stats on one site, film on another, strategy on a whiteboard. The tools are fragmented, and nothing reasons across all three. There is no single assistant that can scout an opponent, watch the tape, and pressure-test a game plan in one place.

CourtVision unifies scout, analyze, and simulate into one grounded workflow, where a finding in one step flows directly into the next.

Problem evidence

Why the disconnect is the real failure mode.

The pain is not that any single tool is bad, it is that nothing reasons across all three. I scoped the build around that gap directly, so the evidence and the demo are the same artifact: one flow that a stack of disconnected tools cannot produce.

Three tools, no shared brain

Scouting stats, game film, and strategy each live in a separate place, so a finding in one never flows into the next.

Plans built on gut

Without a single grounded view of the opponent, game plans lean on memory rather than current-season fact.

You cannot cheaply test a call

A coach cannot ask “what if we press their young guards?” and see it play out before tip-off.

A chatbot is not trustable on the bench

A generic answer with no sources and no confidence signal is not something a coach will stake a game on.

The demo loop was scoped precisely because the disconnect is the failure mode: scout Duke, send the intel to the simulator, call full-court press, and watch the exact turnovers the Scout predicted from Duke’s young guards swing the game. Target users & personas

Three users, three coaching decisions.

CourtVision serves a small, sharply defined set of basketball-staff users, each making a different decision in the same prep-to-bench loop.

The Head Coach

The Analyst

Jobs to be done

  • When I prep for an opponent, give me a grounded, sourced read I can build a plan on, and let me test it before tip-off, so I walk in with a plan I can defend.
  • When I scout an opponent, surface their weakness from current-season data and carry it straight into the game model, so I am not re-keying my own findings.
  • When I call a tactic mid-game, show me how the simulated game reacts, grounded in the scouted weakness, so I stay in control of the decision.

The whitespace

The competitive set is strong at single steps and empty at the seams. CourtVision’s differentiation is the agent-to-agent hand-off plus grounding and human-in-the-loop control, which is exactly what turns scattered tools into one defensible workflow.

Product vision

A coaching staff that can test a plan before it costs a game.

The vision is a single grounded assistant where scouting, film, and simulation reason together, so any coach, not just one with a full analytics staff, can walk to the bench with a plan they have already pressure-tested. The model reads the opponent from real current-season data, carries that read into a game it can run, and reacts to the coach’s calls, while the coach stays the decision-maker. CourtVision is the proof that a generative system can behave like a sourced, rule-bound teammate rather than a free-text chatbot.

Product strategy & positioning

Win on the hand-off, not on any single agent.

The strategy was to make the seam the product. Anyone can ship a scouting chatbot or a game simulator; almost no one makes the scouting flow into the simulation. So I concentrated scope on grounding, the A2A intel store, and the simulation invariants, and deliberately cut everything that did not make that one moment airtight.

The solution

Three agents that pass intelligence to each other.

The same idea, in two voices: the product story for a recruiter, and the system detail for an engineer.

In plain language

One assistant for three coaching jobs.

A coach normally juggles three tools: a stats site to scout the opponent, game film to study them, and a whiteboard to plan. CourtVision puts all three in one place and makes them talk to each other.

The magic moment: the assistant scouts a weakness, then carries that exact finding into a live game simulation. When the coach calls “full-court press,” the simulated game produces the turnovers the scouting predicted, so the plan is tested, not guessed.

Under the hood

Agent-to-agent hand-off across three Gemini agents.

A FastAPI backend orchestrates three agents over a shared in-memory session: Scout (Gemini 2.5 Flash + Google Search grounding, multi-turn memory) returns an answer, sources, confidence, and follow-ups; Game-Tape Analyzer (Gemini 2.5 Pro + vision) breaks down YouTube film by timestamp; Simulator (Flash in JSON mode) emits play-by-play a 2D court renders.

The unlock is A2A: the Scout compiles a tactical brief into a shared intel store the Simulator reads at tip-off, so the game reflects real scouting, not generic play. Deployed on Cloud Run with a React front end.

Key user flows

The flows the product had to nail.

Three flows carry the whole experience, and the hand-off flow is the one the demo lives or dies on.

1 · Grounded scout lookup

Coach asks a plain question → Scout runs Google Search grounding → returns answer, sources, a confidence tier, and follow-ups. Multi-turn, so a follow-up needs no restated context.

2 · Scout-to-sim hand-off

Coach sends the read → Scout compiles a compact JSON tactical brief → the brief is written to the shared intel store → the Simulator reads it at tip-off, so the game reflects the real scouting.

3 · Coach a live game

Coach runs the sim and calls a tactic like full-court press → the Simulator reacts within the invariants → the predicted turnovers appear and swing the game, while the coach stays in control.

Key decisions & tradeoffs

What I chose under the clock.

Two Gemini calls on start

One grounded stat fetch with search, then a separate JSON simulation call with no tools.Tradeoff: an extra call, in exchange for clean grounding and reliable JSON.

Preset coach buttons, not free text

Six tactical presets keep the simulation predictable and the demo crisp.Tradeoff: less open-ended; a deliberate v1 scope cut.

Shared in-memory intel store

The Scout writes a brief the Simulator reads, the A2A pipeline, with zero infra.Tradeoff: not durable across restarts; fine for the hackathon.

Hard simulation invariants

Scores only rise, clock only falls, injured players never appear.Tradeoff: prompt + validation overhead, in exchange for believable games.

Risks & mitigations

Where a generative system can go wrong, and how I hardened it.

Because the whole product is trust, the two places a generative model can fail, ungrounded scouting and a nonsensical simulation, were each hardened directly rather than left to the model’s good behavior.

RiskMitigation
Ungrounded or stale scoutingGoogle Search grounding with a current-season-only instruction, plus a confidence score that drops on thin or conflicting sources
Unreliable JSON from a generative simA dedicated JSON-mode call with no tools, plus a validation layer that rejects any rule-breaking possession before it renders
Open-ended input breaking the demoPreset tactical buttons in v1; free text deferred behind a future approval card
State lost on redeployAccepted for the hackathon scope; durable session persistence is a planned fast-follow
Cost overrun on the budgetFlash on the hot paths, Pro reserved for film, the whole system held under a $25 ceiling
Over-reliance on the AIThe coach stays in control; the AI reacts and never makes a binding call

Learnings & retrospective

What I would carry into the next build.

Scope the moment, not the surface area

The PRD’s real job was deciding what to cut. Concentrating everything on the A2A hand-off, and cutting the box score, win-probability, and free text, is what made the demo airtight on the clock.

Trust is an architecture decision

Sources, a confidence tier, and hard invariants enforced by a validation layer are what let a generative system read as a sourced teammate, not a chatbot. I would design for that from line one again.

Split the calls that fight each other

Grounding and JSON mode pull in opposite directions, so separating them into two calls bought clean grounding and reliable structured output, well worth one extra call.

Make the demo the evidence

Building one live flow that lights up every rubric dimension beat asserting them in slides. I would scope future zero-to-one work around a single provable moment.

If I had more time

I would open up free-text coaching behind a guardrailed approval card, close the loop with a post-game box score and win-probability trace, and persist sessions so a game survives a redeploy, the three honest fast-follows from the v1 scope cuts.
