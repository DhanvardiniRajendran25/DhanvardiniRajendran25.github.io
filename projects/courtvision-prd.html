02 · Executive summary

One workflow, not three disconnected tools.

Coaches prepare across three disconnected tools: stats on one site, film on another, and strategy on a whiteboard, and nothing reasons across all three. CourtVision AI unifies scout, analyze, and simulate into one grounded workflow, where a weakness the Scout finds flows straight into the game the Simulator runs.

The product bet is that agent-to-agent hand-off is the product, not three separate toys. A Scout agent on Gemini 2.5 Flash answers a question like “what is Duke’s biggest defensive weakness this season?” with grounded, sourced facts and a confidence score, compiles that read into a compact tactical brief, and a Simulator agent reads the brief at tip-off so the simulated game reflects the actual scouting. The coach stays in control: the AI reacts to the coach’s calls, it never overrides the bench.

03 · Background & context

Why a multi-agent coaching assistant, and why now.

Coaching intelligence lives in three places at once. Scouting stats live separately from film, which lives separately from strategy on a whiteboard. Coaches spend hundreds of hours preparing across those fragmented tools, and nothing reasons across all three, so game plans are built on memory and gut rather than on a single grounded view. A coach cannot cheaply test “what if we press their young guards?” before tip-off.

CourtVision AI was built and deployed inside a Google Frontiers Hackathon window on a sub-$25 cloud budget, as a single live flow that proves grounded search, agent-to-agent intel sharing, and human-in-the-loop control. Recent grounding tools (Google Search grounding) and reliable structured output (JSON mode) make it possible for a generative model to behave like a sourced, rule-bound system rather than a free-text chatbot, which is what a coach needs to trust it.

Context

The system is integration-ready by design: a FastAPI backend orchestrates three agents over a shared session, so the scout, the analyst, and the simulator are independent surfaces that communicate through a single intel store rather than one monolithic prompt.

04 · Problem statement & evidence

The prep is fragmented, and plans are untested.

Problem: a coach cannot scout an opponent, study the film, and pressure-test a game plan in one place, and crucially there is no tool where one step’s finding flows into the next, so plans are assembled by hand across tools and never tested before tip-off.

  • Scouting stats, game film, and strategy each live in a separate tool, so nothing reasons across all three.
  • Plans are built on memory and gut because there is no single grounded view of the opponent.
  • A coach cannot cheaply simulate “what if we press their young guards?” before the game.
  • A generic chatbot answer is untrustworthy on the bench because it has no sources and no confidence signal.

Who feels it most: the head coach assembling a plan under time pressure, the analyst stitching stats and film together by hand, and the staff who want a plan they can defend rather than one they hope holds up.

Evidence from the build

The demo loop was scoped precisely because the disconnect is the failure mode: scout Duke, send the intel to the simulator, call full-court press, and watch the exact turnovers the Scout predicted from Duke’s young guards swing the game. One flow exercises grounded search, agent-to-agent hand-off, real stats, and human-in-the-loop control at once, which a stack of disconnected tools cannot do.

  • Coach, plan: “How should Auburn attack the first five minutes, given our injuries?” goes to a plan that respects the injured-player constraint and the coach’s style.

06 · Goals, success metrics & non-goals

What success looks like.

North Star metric · TTGP

Time-to-Defensible-Game-Plan (TTGP), how fast a coach goes from a question to a grounded, sourced plan they would actually trust on the bench. It only counts if the answer is cited and the coach stays in control, which is why grounding and human-in-the-loop are metrics, not nice-to-haves.

Goal tiers

  • User goal: run a real coaching workflow, scout to analyze to simulate, and walk away with a grounded plan fast.
  • Business goal: prove a deployable, integration-ready multi-agent system inside the hackathon window and under a $25 cloud budget.
  • Model goal: keep every scout answer grounded in real current-season data with a confidence score, hand intel between agents reliably, and never break a simulation invariant.

12 · Model & prompt strategy

Gemini 2.5, split by job.

The agents differ mostly by their model and prompt contract. The Scout runs Gemini 2.5 Flash with Google Search grounding for fast, sourced reads; the Analyst runs Gemini 2.5 Pro with vision for film; the Simulator runs Flash in JSON mode for reliable structured play-by-play. Two Gemini calls fire on simulation start: one grounded stat fetch with Search, then a separate JSON simulation call with no tools, an extra call in exchange for clean grounding and reliable JSON.

Prompt strategy

  • Scout confidence contract: the prompt forces an answer plus sources, a HIGH / MEDIUM / LOW confidence, and 2 to 3 follow-ups, and instructs the model to lower confidence on thin or conflicting sources. It also forbids building a plan around an injured player.
  • A2A brief contract: the scouting is compiled into a compact JSON object (opponent, key weakness, recommended tactics, injured / unavailable) so the Simulator consumes scouting deterministically rather than re-reading prose.
  • Simulator invariant contract: JSON mode with explicit invariants, scores only rise, the clock only falls, no injured player on the court, react to the latest coach action, with a validation layer enforcing the rules beyond the prompt.
  • Tool separation: grounding tools are attached only to the Scout call; the Simulator call runs tool-free so structured output stays reliable.

Why structured contracts

Forcing sources and a confidence field is what makes the Scout trustworthy: the coach sees the receipts and how sure the model is, which is the heart of TTGP. Structuring the hand-off as JSON makes the brief a contract between agents, so the simulated game reflects the actual read instead of generic play.

13 · UX requirements

Show the receipts, keep the coach in control.

  • Plain-English scouting: a single scout chat; the coach asks normal questions about an opponent, no stat-site navigation required.
  • Sourced answer with confidence: every scout answer shows its sources and a HIGH / MEDIUM / LOW confidence badge so the read is checkable.
  • Send-to-sim hand-off: a clear path from a scouting read to a tactical brief the Simulator will use.
  • Live 2D court: a 2D half-court with a scoreboard and a moving ball renders the play-by-play.
  • Coach controls: preset tactical buttons (continue, full-court press) let the coach intervene; the simulation reacts to the call.
  • Believable game: the scoreboard only rises and the clock only falls, so the game reads as real.
  • Conversational scouting: follow-up questions work without restating the scenario.

Human-evaluation rubric

A coach or domain reviewer scores sampled outputs against fixed dimensions, so “good” is defined before anyone grades.

DimensionDefinition
CorrectnessThe factual claims in the answer match the current-season reality of the opponent.
Groundedness / citationEvery claim is backed by a returned source URL; nothing is asserted without a receipt.
Tactical soundnessThe recommended tactic is a coherent, defensible response to the scouted weakness.
Simulation validityThe play-by-play obeys the invariants and reacts to the coach’s call as a real game would.
SafetyNo plan around an injured or unavailable player; no ungrounded claim presented as fact.

Automated evaluators

  • Citation presence: assert every scout answer carries at least one source URL, or is explicitly low-confidence / abstaining.
  • Invariant validator: the same validation layer that guards production replays the golden set and flags any score drop, clock reversal, or injured player.
  • Schema validity: confirm the Simulator emits well-formed JSON possessions that the 2D court can render.
  • Constraint check: verify the injured / unavailable list in the A2A brief never appears on the simulated court.

LLM-as-a-judge

For the open-ended dimensions that a simple assertion cannot grade, an LLM judge automatically scored scout-answer quality and tactical soundness against the golden answers, so a regression in answer quality surfaces without waiting on a human pass for every change.

Limitations of the judge

The LLM judge is an aid, not the verdict. It carries positional and verbosity bias (it can favor longer or first-listed answers), and it has sport-domain blind spots: it does not reliably know whether a tactic is actually sound for a given matchup. So the judge gates obvious regressions, and a human coach spot-checks a sample of its calls, especially on tactical soundness, before any change ships.

Baseline, target thresholds & regression testing

The baseline to beat is a plain Gemini chat answer with no grounding, no sources, and no confidence signal, the untrustworthy bench answer the product exists to replace. CourtVision has to clear it on every dimension.

  • Segment-level evaluation: quality is read per query type and per agent (Scout, Analyst, Simulator), not as one blended number, so a weak spot in, say, constrained game plans cannot hide behind strong single-stat lookups.

15 · Responsible AI, safety & constraints

Grounded, constrained, and human-led.

  • Human approval: the coach approves and triggers strategy before it is applied; the AI reacts, it never overrides the bench.
  • Grounding as trust: a confidence score and sources ride on every grounded scout prediction, so a thin read is labelled rather than presented as fact.
  • Constrained personalization: the system respects the coach’s style and never builds a plan around an injured or unavailable player.
  • Believability invariants: scores only rise and the clock only falls, enforced by a validation layer, not just by prompting.
  • Public data only: scouting draws on public current-season sources via Google Search; there is no private player data or PII in scope.

16 · Grounding & simulation integrity

Built so a generative system stays believable.

Because the whole product is trust, the two places a generative model can go wrong, ungrounded scouting and a nonsensical simulation, were each hardened directly. Grounding plus a confidence score guards the Scout, and JSON mode plus an invariant-validation layer guards the Simulator, so a bad source or a bad token cannot become an authoritative-looking output.

Why two calls, not one

It costs one extra call per start, in exchange for clean grounding and reliable structured output that the validation layer can then enforce.

17 · Analytics & experimentation

Instrument the bet.

  • TTGP instrumentation: track how long a coach takes from a scouting question to a grounded, sourced plan they would trust, the North Star.
  • Grounding counters: monitor the share of scout answers with real sources and the distribution of confidence tiers, so low-source answers are visible.
  • Invariant violations: count how often the validation layer rejects a possession, so a regression in the simulator prompt is caught early.
  • Coachability signal: track whether a coach intervention (for example a press) measurably changes the simulated outcome, which is the qualitative “it reacted to my call” metric.

21 · Cost, rollout & monitoring

Ship, watch, and extend.

Shipped (v1)

CourtVision AI shipped as a working product deployed live to Google Cloud Run within a $25 credit budget, using Gemini 2.5 Flash and Pro via the google-genai SDK with a React and Tailwind front end: a Scout with grounded search, an A2A intel hand-off, and a coachable JSON-mode Simulator with hard invariants, all in one live flow.

Monitoring & observability

  • Instrument TTGP: time from a scouting question to a grounded, sourced plan.
  • Watch grounding counters: source coverage and the confidence-tier distribution on scout answers.
  • Track invariant-violation rejections so a regression in the simulator prompt surfaces fast.
  • Watch the guardrail metrics: citation-failure rate, hallucinated-stat rate, P95 latency per agent, and cost per session against the budget.

Per-response traceability

Every response is traced with the model version, the prompt version, and the grounding config (which agent, search on or off, JSON mode on or off) that produced it. So when an answer looks wrong, it is clear exactly which model, prompt, and tool setup made it, and a bad change can be pinned to the version that introduced it.

Model & prompt change management

  • Model and prompt versions are pinned and recorded with each response, so a swap is deliberate and auditable, never silent.
  • Any prompt or model change runs the golden set plus the automated evaluators as a gate before it ships.
  • A human spot-check on the LLM-judge calls runs on each release, with the option to roll back to the prior pinned version if a guardrail regresses.
